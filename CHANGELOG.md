# Changelog

## 0.1.42 — 2026-07-22

### Fixed

- **`hub_fw_update.py` gave up on a slow device one step short of the finish** (`skills/_scripts/hub_fw_update.py`). The per-flash `XFER_TIMEOUT` hard ceiling was 20 min, but a marginal-link Leviton/ZEN04 (~−90 dBm) transfers slowly — one hit the ceiling at **93%** and was logged `failed`, yet the device **completed the OTA on its own** moments later and re-interviewed at the target version (confirmed: the very next idempotent run `SKIP`ped it as already-current). The ceiling is only a backstop; the **no-progress watchdog** (abort if `percent` stalls at any level) is the real hang guard, so a slow-but-still-advancing transfer must not be killed at an artificial limit. Raised to 35 min. Fourth and final live-hardened tuning (0.1.39 reboot token, 0.1.40 only-after-failure, 0.1.41 any-node/wide-window canary, 0.1.42 generous ceiling).

## 0.1.41 — 2026-07-22

### Fixed

- **`hub_fw_update.py` canary false-positived "hung" both ways on a busy hub** (`skills/_scripts/hub_fw_update.py`, `rules/firmware-update.md`). The canary refreshed **one** node and waited a tight 30 s for *that* node's `lastTime` to advance. On a 75-LR-node hub a single presence plug often doesn't answer within 30 s even when the radio is fine, so after a (legitimately) failed flash the guard **false-read "hung" and rebooted a healthy hub**, then — while zwaveJS was still interviewing all nodes post-reboot — **false-read "still hung"** and aborted. Verified live: a 0%-stall on node 373 (which transfers nothing, so cannot wedge the radio) triggered a needless main reboot; an independent 4-node probe confirmed the radio was ALIVE the whole time. Fixed: a hung controller freezes **every** node's `lastTime`, so the check now nudges the canary and, over a wider window (75 s), returns healthy if **ANY** node's `lastTime` advances — robust to one quiet/slow node and to a re-interviewing hub. Reboot settle raised to 150 s. This is the third and final guard-tuning fix (0.1.39 reboot token, 0.1.40 only-after-failure, 0.1.41 any-node/wide-window); detection now matches the real hung-controller signature (all nodes frozen) instead of one node's timing.

## 0.1.40 — 2026-07-22

### Fixed

- **`hub_fw_update.py` rebooted the hub after every *successful* flash** (`skills/_scripts/hub_fw_update.py`, `skills/firmware-update/SKILL.md`, `rules/firmware-update.md`). The canary radio-health probe ran after *each* flash, but a just-flashed device reboots and re-interviews, which busies the radio — so the canary's node didn't answer inside its 30s window and the guard **false-concluded "hung" and rebooted the hub after a clean flash** (caught live: node 312 verified 3.6, then an unnecessary reboot). A batch would thus reboot the hub once per device. Fixed: run the canary/reboot recovery **only after a `failed` flash** — a verified success already proves the radio transmits; only a failed/stalled flash can have hung it. Also hardened `node_lasttime` to tolerate a momentarily-unreachable hub (mid-reboot) instead of crashing the run, and the `rebooted` counter is now incremented. Silver lining: the false trigger exercised the (0.1.39-fixed) reboot recovery end-to-end — HUNG → reboot → "hub back up" → re-check — confirming detection *and* recovery now both work.


## 0.1.39 — 2026-07-22

### Fixed

- **`hub_fw_update.py` auto-reboot recovery could not get a management token** (`skills/_scripts/hub_fw_update.py`). `GET /hub/advanced/getManagementToken` returns a **bare token string**, not JSON — the reboot path read it with `json.load`, which threw `Expecting value: line 1 column 1`, so when a failed OTA hung the controller the guard detected the hang but **never actually rebooted**. Caught live: a ZEN76 flash hit `stage=FAILED`, the canary correctly saw the radio hung, then the reboot silently no-op'd and the hub stayed hung until a manual reboot. Fixed to read the token as text (`.read().decode().strip().strip('"')`). The detection half (watchdog + canary) worked; only the recovery half was broken.


## 0.1.38 — 2026-07-22

### Added

- **Firmware-update skill: safe Z-Wave OTA batch-flashing** (`rules/firmware-update.md`, `skills/firmware-update/SKILL.md`, `skills/_scripts/hub_fw_update.py`, closes #64). Grounded live on 2.5.1.132 (C-8 Pro, zwaveJS) flashing ~67 Zooz/Leviton devices across two hubs. Uses the hub's **native** Device firmware updater (`/hub/zwave/deviceFirmware/{files,details,available,start,progress}` + `/hub/fileManager/upload/firmware`) — flashes **in place** over LR+S2 with no driver swap; the community driver-swap updaters stall on LR/S2 ("Please wake up your sleepy device" on a *mains* device = wrong tool). The load-bearing finding, learned twice the hard way: a **failed OR stalled** OTA hangs the **entire** zwaveJS controller — the hub returns `success:true` but transmits nothing, freezing every Z-Wave node at once while **Zigbee stays up** (the diagnostic tell), the Hub-Mesh peer still reads `active/reachable`, and downstream it **staleness-poisons every lux/temperature-gated automation** (a "Motion at Dark" rule fired lights in daylight off a frozen illuminance sensor). A big fleet hides it (successes after a failure re-kick the radio); the trailing/only failure is the dangerous one. `hub_fw_update.py` carries the guards, **all required** (a canary-only guard *missed* a mid-transfer stall): a **no-progress watchdog** (abort if `percent` stops advancing at ANY level, not just 0%), a **canary** radio-health probe between flashes that **reboots + re-checks** on a hang, and an **RSSI floor** (default −95 dBm) that skips hang-prone floor-signal nodes (`--flash-weak` to force, attended only) — plus idempotent skip, a circuit breaker, and `--wait-pid` to chain one radio. `SKILL.md` walks discover (`/device/fullJson` → `device.data.firmwareVersion`) → **vendor-latest, NOT auto-discovery** (the Z-Wave JS service lagged ZEN04 2.30 vs shipped 2.60, missing the 2.50 SDK 7.19→7.24 S2/SPAN fix, and offered a Springs shade a bogus downgrade) → hardware-revision-correct image (700 vs 800LR) → stage (upload once per model, reusable) → flash → verify (hub caches the old version until the post-reboot re-interview). Recovery: `getManagementToken` → `/management/reboot` clears the hang; nothing bricks (`nodeState OK`), a stalled node sometimes completes post-reboot.

## 0.1.34 — 2026-07-22

### Added

- **Room Lighting can group *shades* (staff-endorsed) — `Act==Off` is a one-position preset, not a broken toggle** (`rules/room-lighting-shades.md`, closes #57). An automated audit of a live hub flagged two RL instances (`Alice Office Shades`, `Amir Bedroom Shades`, both `0/0`) as bugs — "activator has no open path, Act==Off position." False positive: Room Lighting drives `capability.windowShade`, and repurposing an RL activator device as a shade group is a pattern Hubitat staff (bravenel) endorse (community thread `t/new-app-room-lighting/93098`). In the per-device capture table the **Position (Act)** column is the position sent on activation (**100 = open, 0 = closed**) and **Off** is the deactivation position, so an RL shade group is a position preset, not an open/close toggle — `Act==Off` is a valid one-position group (`0/0` = always close, `100/100` = always open), composed into a nested activator hierarchy where opening is done by a *different* group. New always-on rule: don't flag a shade RL instance as broken merely because `Act==Off` or because "no path opens the shades in this instance" — verify intent from the hierarchy and `position 0 == closed` before calling it a bug.
- **RL activation-options switch guards *are* automatable — SumoSelect enums + an inline Vue picker, not `#deviceListModal`** (`skills/_reference/playwright-ui.md` 26, `rules/ui-automation.md`, closes #56). The "Disable/Re-enable Activation when a switch turns on/off" guard on a Room Lighting instance's *Activate Lights Options* sub-page first looked un-automatable (the switch device-picker seemed to be a Vue-3 modal that wouldn't mount on a Playwright click), then cracked end-to-end on #918/#921 (2.5.1.x, 2026-07-21): the picker is **not** `#deviceListModal` (a dead empty shell) but an **inline Vue list** rendered under the device button. Working recipe (via `browser_run_code_unsafe`, gotcha 23): the `settings[onDisable]`/`settings[onEnable]` enums are **SumoSelect** that commit via `submitOnChange` on dropdown **close** (open `.CaptionCont` → click `li.opt` → close `.CaptionCont`), which AJAX-persists the enum and reveals the picker; then real-click the `data-elemname` button, **filter with real keystrokes** (`page.keyboard.type` — `locator.fill()` doesn't trigger the Vue filter), **check the row by coordinate** (a label-locator click auto-scrolls and collapses the dropdown), **click `Update` by coordinate** (a `div.mdl-button`, not a `<button>`), Done up the chain, verify via `configure/json`. Both `switchesD`/`switchesOE` become required once their enum is set, so a half-set guard makes the config page self-reject on load; revert by clearing both enums.
- **UI-fired requests you can replay — Z-Wave `nodeRemove`, `runmethod` device command, app enable/disable** (`skills/_reference/endpoints.md`, closes #50). Several operations documented as "UI-only" are ordinary HTTP requests the UI fires; drive the UI once with Playwright, read the request, replay it directly thereafter. From migrating the Devices hub to the zwaveJS backend and sweeping orphan ghosts (C-8 Pro, 2.5.1.x, zwaveJS, Hub Security off): `POST /hub/zwave/nodeRemove` (`zwaveNodeId=<decimal>`, no CSRF) force-removes a **FAILED** Z-Wave orphan — verified 23× live, each confirmed by census diff, async so poll `/hub/zwaveDetails/json` for absence — and is guarded to FAILED orphans with no bound `deviceId` (behavior on a healthy node is untested; never POST a real device id). `POST /device/runmethod` (`{"id","method","args"}`) sends a device command **without Maker API** and was verified flipping a Z-Wave `nodeState` FAILED→OK (the "flash a stale device to wake it" primitive). `POST /installedapp/disable` (`{"id","disable":<bool>}`) enables/disables any app instance (verified 2026-07-21 from #56's work).
- **Changing an input's `type` between versions strands the stored setting; `removeSetting` is deferred** (`rules/groovy-gotchas.md`, `rules/app-lifecycle.md`, closes #36). From a live in-place upgrade of `jbaruch/hubitat-ecobee-lite` (its #29/#30, verified 2.5.1.128): Hubitat keeps a setting's original type across a code change, so an input redeclared from `type: "capability.thermostat"` to `type: "enum"` still hands the new code a `DeviceWrapper` from `settings.thermostat`, and a String-typed method throws `No signature of method … DeviceWrapper` — which took down the config page and would take down `initialize()` and every handler while the instance stayed subscribed. Invisible until upgrade: a fresh install and the offline test harness both store the new type correctly, so CI and a re-install stay green while the in-place upgrader is the one who breaks. `groovy-gotchas` gains the trap after "Null device inputs": prefer remove-and-recreate, or if in-place must survive consume defensively by type — branch on `instanceof String` for a scalar id (never `findAll`, which iterates a String's characters and mangles the id; reserve it for multi-select Lists), detecting the stale value with `getObjectClassName` (not sandbox-banned `getClass`). Plus its sibling: `removeSetting` is **deferred** — it does not update the in-memory `settings` map in the same execution, so it is cleanup for the next wake, never a same-render crash-guard (the guard is the defensive read). `app-lifecycle` gains a one-line note that `updateSetting`/`removeSetting`/`clearSetting` exist and that `removeSetting` is deferred.

## 0.1.25 — 2026-07-20

### Added

- **`hubStartupHandler()` skips `initialize()` — guard `state.*` or it NPEs on boot** (`rules/app-lifecycle.md`, closes #47). A shipped-app failure surfaced by Error Monitor: `java.lang.NullPointerException: cannot invoke method keySet() on null object (hubStartupHandler)`. The app set up its state maps in `initialize()`, but `hubStartupHandler()` runs on hub startup **without** routing through `installed()`/`updated()`/`initialize()`, so on a boot where the map was still null a shared `reconcile()` reading `state.activeSince.keySet()` blew up. The same trap catches any entry point the platform can invoke before the first Done or after a state reset — a subscribed event, a scheduled job. `app-lifecycle` gains a "Handlers that skip initialize()" section next to the reinitialize idiom: put state-map setup in a small `ensureState()` helper called from every entry point that reads it (or null-guard at the read site), and test it with a spec that drives the startup path on a never-initialized instance and asserts no NPE — it fails before the guard, passes after.

## 0.1.24 — 2026-07-19

### Fixed

Two runtime couplings moved with them: `hub_lint.py`'s `_REF` (which reads `allowed-imports.txt` / `capabilities.json` from a sibling dir) and the mesh eval generator's `sys.path` import of `hub_mesh`. CI, `pyrightconfig.json`, and `.tesslignore` paths updated; tests stay dev-only via the relocated ignore glob. The `skills:`/`rules:` manifest declarations are unchanged — the assets just live where the allowlist can see them now.

## 0.1.16 — 2026-07-17

### Added

- **A value attribute's timestamp is not a liveness signal** (`rules/state-vs-attributes.md`, `skills/debug`, closes #34). The read-side consequence of event change-filtering, from a real watchdog that flapped `DEAD` → `recovered` for 90 minutes and fired nine false alerts against a flawless cloud integration on a C-7 running 2.5.1.128 — the same hub and app as #25, one day later. The rule already carried the writer-side line (*"the platform filters events whose value did not change"*); what was missing is what that means for a reader. Unchanged values emit no event, so `currentState(attr).date` answers "when did this value last change" and a healthy source reporting a steady value is byte-for-byte indistinguishable from a dead one. The measurement is the argument: `temperature` gaps ran 11, 1, 6, 33, 27, 36, 18, 27, 12 min — house thermal inertia — while the poll cadence held every ~3 min, flawless. A 15-minute limit sits mid-distribution, so it oscillates; the watchdog was measuring the house and reporting it as a dead integration. ### Changed

- **A status is not a fact** (`skills/debug`). ## 0.1.11 — 2026-07-16

### Added

- **Route fan-in: a repeater's blast radius** (`zwave.route_fan_in`, closes #22). `hub_mesh.py` ranked and flagged per-node and never asked how many nodes depend on a single repeater. On the grounded hub, **12 nodes route through one repeater the hub has never heard from** — 12 devices whose path runs through something there is no evidence is alive. `never_heard` has surfaced that repeater since 0.1.7, listed identically to a silent leaf in a spare bedroom, and the difference between them is twelve nodes. Nothing in the output said so. It does now: each node carries a `dependent_count`, so a `never_heard[]` entry states its own blast radius inline where the reader already looks, and `route_fan_in.load_bearing_concerns[]` crosses fan-in with the repeater's own health (`never_heard` / `FAILED` / weak-signal / packet errors), ranked worst-first.
- **Fan-in is context, not a fault — and reaches neither counter.** The model is `shared_device_count` on a hub-mesh peer, which informs a re-add's blast radius without ever being a finding of its own. A repeater carrying 12 nodes is a **normal, healthy mesh**; counting repeaters would warn about healthy hubs and teach the reader to ignore the number. Every repeater fan-in marks as a concern is *already* counted through the flag it mirrors — the never-heard repeater warns once, via `never_heard` — so counting it again would double-count one node twelve ways. Fan-in changes what an existing flag **means**, not how many there are. Ranked, never thresholded: Hubitat publishes no "too many dependents" cutoff and this invents none. **Classic mesh only** — LR is a star with no repeaters, so LR nodes are excluded and carry `dependent_count: null`, the question not applying being distinct from zero.
- **Fresh dependents are evidence a flagged repeater still relays.** A repeater cannot be dropping everything while a node behind it reported a minute ago — the rule and skill now say to read the dependents' freshness before blaming the hop. The `mesh-health` eval fixture is built to exercise exactly this: all three of its repeaters carry a fresh device alongside frozen ones, so `route_fan_in` actively refutes "the repeater is dropping traffic" instead of merely not raising it.
- **Incoherent routes are surfaced, never guessed at.** A coherent route runs hub (`01`) → node, through classic-mesh hops only; anything else is a stale or malformed record, and inferring which hops are repeaters out of it would invent dependents. Those land in `route_fan_in.anomalies[]`, and a hop naming a node absent from `nodes[]` is tagged `unknown_node` — data gaps, not mesh faults, in the shape `unparsed_timestamps` already established. This is not hypothetical: the pre-0.1.10 fixture carried exactly that incoherence on 20 of 21 nodes, and fan-in would have read it as one shade repeating for the entire mesh. **A route naming a Long Range or reserved-range id as an intermediate hop is incoherent by the same standard** — a star node relays nothing, so the hop is an anomaly rather than a repeater discovered. Routes are validated before any counting: a route rejected halfway would leave its earlier hops credited with a dependent from a path just called impossible.

- **Two UI routes went undocumented.** `reference/endpoints.md` now carries them: the Settings page is **`/hub/edit`** (`/hub/settings` 404s), and `/installedapp/direct/<builtInAppType>` opens a built-in app by redirecting to a *transient* instance at `/installedapp/configure/<newId>/mainPage` — it takes the next app id, and its **Cancel** discards it (`statusJson` then returns `{}` and it never reaches `/hub2/appsList`). Swap Device is reached this way, so it is a built-in **app**, not a device page. Swapping a device's references is added to the UI-only operation list in `rules/ui-automation.md` and `reference/playwright-ui.md`.
- **A Vue-wrapped `<select>` ignores `browser_select_option`** (`reference/playwright-ui.md`). On Settings → Swap Device, selecting the "old" device that way sets the native select's value while the app never reacts — the dependent "new" picker stays disabled and empty, so the value just written reads back as though it took. The existing no-synthetic-events rule held; the read-back trap is now named.

## 0.1.7 — 2026-07-16

### Added

- **The command path is now a first-class axis of mesh diagnosis.** Grounded on a live outage: every Z-Wave *and* Zigbee actuator on one hub (22 shades, lamps, an outlet) went dead for 13.7 h while sensors kept reporting — and `hub_mesh.py` said `critical: 0, warnings: 0`. The cause was not a radio at all. The hub's hub-mesh peer record for its companion hub still held an IP from the owner's *previous house's subnet*, so commands crossing the mesh were accepted, logged by the app, and silently dropped. The hub reported that dead peer as `active: true, offline: false, warning: null` with `lastActive` ticking every few seconds. No radio metric can see this class of fault, so `hub_mesh.py` now analyzes `GET /hub2/hubMeshJson` (newly documented) beside the two radios: peers are probed for reachability and identity (`hubUID` in `/hub/details/json` is the same identifier as `hubId`, verified across three hubs — so a reassigned address is caught as well as a dead one), and `hub_mesh.problems[]` rolls into `summary.critical`. Replaying the outage's captured bytes through the new code yields `critical: 1 / peer_unreachable` where the shipped code said all-clear. Each problem carries `shared_device_count`, the blast radius of a re-add — the asymmetry mattered live: the broken link shared 3 devices (cheap to re-add) while the *other* direction shared 148, whose removal would have unbound every app using them.
- **Z-Wave node staleness** (`age_seconds`, `stalest`, `never_heard`), the Z-Wave counterpart of `zigbee.stalest`, and the signal that actually cracked the outage. `lastTime` means "when the hub last heard this node"; for a device that only transmits *after* being commanded, that is "when a command last landed". Ranked, never flagged — silence is not a fault, and reading it needs context the script does not have. `never_heard` catches nodes the hub has never heard at all, which are reported `nodeState: OK` and pass every FAILED-keyed check (6 live on the Devices hub, including a repeater carrying 12 nodes' routes).
- **The actuator-vs-reporter split** in the `mesh-health` skill and the `zwave-zigbee-mesh` rule — the reasoning the outage turned on, and the diagnostic worth more than any single flag. Reporters (lock, motion, presence) transmit on their own, so fresh means the radio works; actuators (shade, outlet, lamp) transmit only when commanded. Reporters fresh + actuators frozen ⇒ the *command* path is broken, not the radio — and when it spans both radios at once, no radio fault can explain it, because two radios do not fail in the same second. A cluster of actuator timestamps in one narrow window is the last moment commands worked (here, a 53-second sunset routine, the outage's high-water mark).

### Fixed

- **`mesh-health` reported a confident false all-clear.** Step 2 said that if `summary.critical` and `summary.warnings` were both 0, "report the mesh looks healthy and finish" — so against 22 dead actuators the skill was engineered to stop the operator looking. Zero counters no longer license an all-clear: a new Step 4 splits the staleness rankings by actuator vs reporter, and only a run that survives *that* may report healthy. The rule and skill now open by stating that a clean radio is not a working device, since every radio metric describes a path commands do not travel.
- **A Zigbee-only run no longer warns about Z-Wave timestamps** (PR #18 review). `/hub/details/json` exists only to date Z-Wave's naive stamps — Zigbee's `lastActivity` carries its own offset — but it was fetched unconditionally, so `--radio zigbee` against a firmware lacking it raised a `fetch_warning` about node ages that run never computes. Since the skill reads any fetch warning as a blind axis blocking an all-clear, an irrelevant warning is a *false* blind axis. The fetch is gated on the Z-Wave path.
- **A peer answering 200 with JSON but no `hubUID` read as identity-verified** (PR #18 review) — `hubId: None` with `error: None` is the one shape that must never look checked-and-fine. It now carries an explicit identity-unverified error like the other unverified cases.
- **The peer probe no longer conflates reachability with identity** (caught independently by both PR #18 reviewers). `probe_peer` treated any `HubError` from `/hub/details/json` as `reachable: false` — but that endpoint is itself undocumented and version-sensitive, so a peer answering HTTP without it would have rolled a false `peer_unreachable` **critical** against a perfectly healthy peer. The two are now separate questions: `reachable: false` is reserved for a connection-level failure (nothing answered), while an HTTP response lacking or not parsing `hubUID` is reachable-with-identity-unverified, which `analyze_hub_mesh` reads as no evidence rather than as a mismatch. Re-verified against the real dead address from the outage: still `critical: 1 / peer_unreachable`.
- **Script runtime floor raised to Python 3.9+** (`scripts/README.md`), which `zoneinfo` requires and the previous documented 3.8+ contract would have broken at import. Guarding the import and degrading to UTC was the alternative and is worse than a clear ImportError: it silently overstates every zwaveJS node's age by the hub's UTC offset — precisely the bug the zone lookup exists to fix. Python 3.8 reached end-of-life on 2024-10-07 and CI runs 3.11. The stale `3.8-compatible` comment in `hub_mesh.py` is corrected to match.
- **Auxiliary endpoints no longer sink the radio diagnostics** (caught by the OpenAI policy reviewer on PR #18 as an `error-handling` Graceful Fallback violation). `/hub2/hubMeshJson` and `/hub/details/json` are undocumented and version-sensitive, and the first cut fetched them in the same fatal `try` as the radios — so a hub or firmware lacking either returned nothing at all, though `analyze()` already models `hub_mesh=None` and `naive_zone(None)` already has a fallback. Non-zero exit is now reserved for the requested radio endpoints; an auxiliary miss degrades the affected key to null and lands in a new `fetch_warnings[]` (and stderr) naming its own `consequence`. Degraded loudly, never silently: a null `hub_mesh` leaves a command-path fault unruled-out, so the skill treats a fetch warning as a blind axis rather than an all-clear. `main()` now takes `transport=` for injection, matching `hub_device_usage.py`.
- **`never_heard` misclassified unparseable timestamps** (caught by Copilot on PR #18). It keyed on `age_seconds is None`, which `parse_ts` also returns for a `lastTime` that is *present but malformed* — inventing "the hub has never heard this node" out of a parse failure, and inflating warnings now that `never_heard` counts. It keys on the raw field's absence; a present-but-unparseable stamp surfaces separately as `unparsed_timestamps[]`, a data anomaly rather than a mesh fault.
- **Flagged Z-Wave warnings never reached the rollup.** `analyze()` counted only `failed` and `packet_errors`, so `weak_signal_heuristic` (and the new `never_heard`) were computed, returned, and dropped from `summary.warnings` — a hub with routes flagged at/below the noise floor still rolled up `warnings: 0`, which the old Step 2 read as healthy. A second false all-clear, independent of the hub-mesh one and the same bug class. Found while verifying the claim that the legacy RSSI floor went unflagged: it does not — the floor works and had flagged two live nodes at `-4dB` and `0dB`; the rollup was throwing their verdict away. Live on the Devices hub, `warnings` goes 0 → 8.
- **Naive `lastTime` stamps were read as UTC, aging every zwaveJS node by the hub's offset** — a third backend trap beside the `lwrRssi` scale. The legacy backend stamps `lastTime` with an explicit `+0000` (true UTC); the zwaveJS backend emits a *naive* stamp in the hub's *local* zone. Measured live: a 70-second-old node read as 5.02 h on `America/Chicago`. Left unfixed, the staleness feature above would have been fiction on half the fleet. `parse_ts` now takes the zone, fetched from the hub's own `timeZone` rather than assumed from whatever machine the analyzer runs on.
