# Changelog

## 0.1.13 — 2026-07-16

### Added

- **Snapshot `ref`s resolve to the wrong element on Hubitat's MDL controls, and say nothing about it** (`reference/playwright-ui.md` gotchas 10–15, closes #30). From installing a user app instance end-to-end on 2.5.1.128 — 2 device radios, 25 contact-sensor checkboxes, 5 enum dropdowns, Done. The expensive one: `browser_find` returned a `ref` for a picker's "Update", and the click it generated was `page.locator('.w-full.flex.flex-row').first().click()` — a **container**. No error, the picker looked right, and `settings[thermostatA]` stayed `""`. The cause is structural rather than incidental: Hubitat's controls are MDL **`<div>`s, not `<button>`s**, so the accessibility snapshot labels a generic wrapper and the generated selector takes the wrapper's class soup. Every technique resting on `ref` or `getByRole('button')` inherits it.
- **Tag-then-click, the pattern that makes the existing rule usable.** Walk up from the hidden input — the only stable identifier on the page — tag the real control, click the tag for real. This threads the needle the "act with real `browser_click`" rule leaves: assigning an `id` inside `browser_evaluate` is **not** the forbidden thing, since the ban is on synthetic `element.click()`, not on DOM tagging. Precise target, genuine event. The picker's Update is `div.device-save`, **one per device input on the page**, so a bare selector hits the wrong picker — scope by walking up from `input[name="settings[<name>]"]` and filter on `offsetParent !== null`, since only the open picker's Update is visible.
- **The hidden input is the commit signal, and it is the cheapest check available.** `input[name="settings[<name>]"]` is `""` until the picker's Update commits and holds a comma-separated id list after — which caught the `ref` failure immediately. The `verify every mutation` rule was already right; this is the concrete signal for device pickers. Order is **selection order, not sorted** — compare as a set. This also sharpens gotcha 2 rather than contradicting it: Update commits into the *form*, **Done** persists to the *hub*; the two are different events and only the second was documented.
- **`submitOnChange` device inputs gate the sections below them, so ordering is load-bearing.** Dependent sections re-render only after the picker's **Update** commits — not when the option is clicked — so controls built from the selected device's data do not exist yet and there is nothing to `selectOption`. Commit the device input first, then fill what it gates. Selections already made downstream survive a later re-render (verified: five dropdown values intact after a second picker committed).
- **An app config page is a transient instance — protect it with a second tab.** Opening a user app lands on `/installedapp/configure/<newId>/mainPage` with a real id, but nothing persists until Done and the form carries `_cancellable: false`. Never navigate the configuring tab; do work elsewhere in a second tab and re-verify on return (config survived intact). Same family as the built-in app's transient instance discarded on Cancel, already in `reference/endpoints.md`.

### Fixed

- **A documented mechanism that did not survive contact with the DOM** (`rules/ui-automation.md`, `reference/playwright-ui.md`). Both files stated that MDL/Vue pickers "leave `input.checked` at `false`". On 2.5.1.128 the property **agreed with the class every time** — on a device radio and on 14 multi-select checkboxes. The earlier observation is real too: reading the property once made 15 selected members look unselected and nearly wiped them. Both being real *is* the finding, so the claim is now what the evidence supports — `input.checked` is **unreliable and may or may not track**, depending on element and platform version. The guidance is unchanged and was never in doubt: `label.is-checked` is the safe superset in every case measured. This is worth more than the DOM detail it corrects. A reader who sanity-checks a stated mechanism and finds it false has no way to tell which of the file's *other* claims still hold, so an over-specific mechanism is a liability that a hedge is not — the same reason #25's version scope ships as correlation rather than a boundary. Fixed at the mechanism, not by deleting the advice.

## 0.1.12 — 2026-07-16

### Changed

- **The `device-migration` swap-blocked scenario was measured and not shipped** (closes #23). `plugin-evals` makes lift an **admission gate, not a curation afterthought**, so the candidate was built and run before being committed: **baseline 100%, with-context 100% — zero lift** (`deepseek-v4-flash`, 3 runs each, against `hubitat-dev@0.1.11`). It does not land, and `evals/README.md` records the measurement so the same candidate is not re-proposed as an obvious gap. The suite stays at three scenarios.
- **The predicted baseline failure simply did not happen.** #23 expected the intuitive ladder (swap → virtual → manual) to bait a floor model into routing a child device through a virtual hop that cannot work. It did not: the model read the `/hub2/devicesList` tree, found the device nested in its parent's `children[]`, concluded child devices are not offered for swap, and went straight to the manual path — never raising the hop. It named the affected apps and both dashboards from the usage capture unaided. #23 called this retire-risk in advance and called it correctly.
- **The lesson generalizes past this scenario.** The issue required the fixture make the device's parentage *genuinely inferable from the data*, or the diagnosis would be a guess — and anything genuinely inferable, a capable model infers. That tension is the scenario's whole story: `plugin-evals` cause 1 (coincidence with universal competence), where retiring is the prescribed outcome rather than a construction failure. One fix was considered and rejected: tightening the virtual-hop criterion to demand an *explicit* rebuttal would have manufactured ~34 points of lift, since the with-context run rejects the hop by name while the baseline never raises it — grading verbosity rather than correctness, when both answers walk the owner to the same correct migration. The rule anticipates that too: *"do not rewrite toward 'testing reasoning' if baseline already reasons to the same outcome."* The **parent-app** child (not indented anywhere, its only tell a `parentApp` row) remains worth measuring, since the tree inference that carried the baseline is unavailable there — but it is **not grounded yet**, the live 2.5.1.128 verification having covered parent-*device* children only, and inventing grounding to make an eval work is how a fixture starts lying.

## 0.1.11 — 2026-07-16

### Added

- **Route fan-in: a repeater's blast radius** (`zwave.route_fan_in`, closes #22). `hub_mesh.py` ranked and flagged per-node and never asked how many nodes depend on a single repeater. On the grounded hub, **12 nodes route through one repeater the hub has never heard from** — 12 devices whose path runs through something there is no evidence is alive. `never_heard` has surfaced that repeater since 0.1.7, listed identically to a silent leaf in a spare bedroom, and the difference between them is twelve nodes. Nothing in the output said so. It does now: each node carries a `dependent_count`, so a `never_heard[]` entry states its own blast radius inline where the reader already looks, and `route_fan_in.load_bearing_concerns[]` crosses fan-in with the repeater's own health (`never_heard` / `FAILED` / weak-signal / packet errors), ranked worst-first.
- **Fan-in is context, not a fault — and reaches neither counter.** The model is `shared_device_count` on a hub-mesh peer, which informs a re-add's blast radius without ever being a finding of its own. A repeater carrying 12 nodes is a **normal, healthy mesh**; counting repeaters would warn about healthy hubs and teach the reader to ignore the number. Every repeater fan-in marks as a concern is *already* counted through the flag it mirrors — the never-heard repeater warns once, via `never_heard` — so counting it again would double-count one node twelve ways. Fan-in changes what an existing flag **means**, not how many there are. Ranked, never thresholded: Hubitat publishes no "too many dependents" cutoff and this invents none. **Classic mesh only** — LR is a star with no repeaters, so LR nodes are excluded and carry `dependent_count: null`, the question not applying being distinct from zero.
- **Fresh dependents are evidence a flagged repeater still relays.** A repeater cannot be dropping everything while a node behind it reported a minute ago — the rule and skill now say to read the dependents' freshness before blaming the hop. The `mesh-health` eval fixture is built to exercise exactly this: all three of its repeaters carry a fresh device alongside frozen ones, so `route_fan_in` actively refutes "the repeater is dropping traffic" instead of merely not raising it.
- **Incoherent routes are surfaced, never guessed at.** A coherent route runs hub (`01`) → node, through classic-mesh hops only; anything else is a stale or malformed record, and inferring which hops are repeaters out of it would invent dependents. Those land in `route_fan_in.anomalies[]`, and a hop naming a node absent from `nodes[]` is tagged `unknown_node` — data gaps, not mesh faults, in the shape `unparsed_timestamps` already established. This is not hypothetical: the pre-0.1.10 fixture carried exactly that incoherence on 20 of 21 nodes, and fan-in would have read it as one shade repeating for the entire mesh. **A route naming a Long Range or reserved-range id as an intermediate hop is incoherent by the same standard** — a star node relays nothing, so the hop is an anomaly rather than a repeater discovered. Routes are validated before any counting: a route rejected halfway would leave its earlier hops credited with a dependent from a path just called impossible.

## 0.1.10 — 2026-07-16

### Added

- **The `mesh-health` fixture generator now lives in the repo** (`evals/mesh-health-command-path-not-radio/generate.py`, closes #24). 0.1.8 shipped a fixture that was *generated* — synthetic raw hub JSON pushed through the real `analyze()` with an injected clock — and that property is the whole reason the scenario means anything: a **false all-clear** is its premise, so a `summary {critical:0, warnings:0}` that an author simply typed would beg the question the scenario exists to pose. The generator that produced it lived in a scratch directory and is gone, which was the issue's prediction and is now just a fact. Rebuilding it found the drift that a real round-trip would never have allowed: `hub_mesh` carried a hand-written `peers[].probe.{reachable, identity_match}` that `analyze_hub_mesh()` **has never emitted** — it reports `reachable` / `probed_hubId` / `probe_error` flat on the peer, plus a top-level `probed` — so the eval was quietly teaching a field path that does not exist on any hub. `analyze()` takes `hub_mesh` already-analyzed and passes it straight through, so nothing was ever in a position to catch it. Of the three options the issue laid out, this is #1: co-located with what it generates, `.tesslignore`d out of the published plugin, and deliberately **not** a `scripts/` module — `testing-standards` would rightly demand a production test burden for a one-fixture authoring tool. It carries a `--check` mode, and CI runs it across every `evals/*/generate.py`, because a generator nobody replays is the scratch directory again with extra steps.
- **A fixture-hygiene CI gate** (`ci.yml`). `--check` regenerates in memory and diffs against the committed fixture, failing the build — with the regen command in the message — when they part ways. It loops over every `evals/*/generate.py` rather than naming one scenario, so the next generator is gated the moment it lands. Pyright's zero-findings gate widens from `scripts/` to `scripts/ evals/` so the generator is type-checked like everything it imports. Announced in scope rather than smuggled, per `ci-safety` Hands Off CI Config. Without it the `--check` mode would be a check nobody runs, which `language-diagnostics` correctly calls no gate at all — the scratch directory again, with extra steps.

### Fixed

- **The fixture's routes were fiction, and #22 was about to read them.** Every node carried `route: '01 -> 0A'` regardless of its own id — a copy-paste artifact of the lost generator. For node 10 (`0x0A`) that is a valid direct route; for the other twenty it claims the path *terminates at a different node*. Inert while nothing parsed routes, and route **fan-in** analysis is next. Routes are now coherent: three repeaters (`Extender Hall` / `Cellar` / `Gallery`) relay for the nodes whose routes name them. The topology is deliberately **decorrelated from the freeze** — every repeater carries at least one *fresh* device alongside frozen ones, so no repeater can be read as the cause. That is load-bearing rather than decorative: a repeater carrying only frozen nodes would hand the reader exactly the plausible radio explanation the scenario is built to deny, and fresh devices behind the same repeater rule it out *on the evidence* instead of by the fixture's silence. Two Z-Wave LR nodes (id ≥ 256, `neighbors: 0`) join for the same reason — so fan-in is shown to ignore a star rather than merely never meeting one.
- **`routeChanges` was `0` on a `zwaveJS` backend**, where the hub reports the literal `N/A` — the fixture contradicted this plugin's own `rules/zwave-zigbee-mesh.md` ("The backend split"). It now parses to `null` the way a real zwaveJS hub's does.
- **The fixture was missing the top-level `hub` key** that `main()` stamps onto every real run.

Every element of the scenario's ground truth is preserved **byte-identically** through the regeneration — same `summary`, same stalest ten, same ranked lists, same Zigbee ages, same 21 nodes visible in the output — so `criteria.json` needed no edit, which is the strongest available evidence that the rebuild is faithful rather than merely plausible. The five nodes added to keep the topology coherent stay *invisible* by construction (fresh, fast, strong), and the generator asserts that on every run: weakening one would silently push a graded node out of a ranked list and quietly invalidate the measurement.

## 0.1.9 — 2026-07-16

### Added

- **A `device-migration` skill, and the Swap Device path this plugin never mentioned.** Replacing hardware strands every app reference on a new device id, and until now the only answer here was to re-select device by device — `rules/device-lifecycle.md` prescribed exactly that, and the word "swap" appeared nowhere in the repo. Hubitat's **Settings → Swap Device** re-points every app in one action. The skill reaches for it first and, when it is blocked, picks the fallback by **why** it is blocked rather than trying each in turn. That branch is the point: the intuitive ladder (swap → virtual → manual) routes through a virtual hop that **provably cannot work** for a child device, since the final swap still has to target the child, which stays ineligible however many hops precede it. Verified live on 2.5.1.128 (C-8 Pro): of 151 top-level devices **all 151** were offered in the swap list and **all 5** child devices (a Zooz power strip's outlets) were absent — the exclusion maps exactly onto the parent/child relationship, and the page says so: *"Most child devices are not swappable and are not listed here."* The virtual hop keeps the two jobs it can do: **bridging** a capability gap (the "new" list is filtered to at least one overlapping capability, and stays disabled until an "old" device is picked) and **parking** references when the old device must be excluded before the replacement can pair — a deleted device cannot be swapped from. Two hazards the UI hides are surfaced: the swap is **bidirectional** (apps already using the *new* device are moved onto the *old* one), and Hubitat scopes the feature to "all apps" while claiming nothing about **dashboards**, so dashboards get verified by hand rather than reported as migrated. `device-removal` Step 6 now hands off here instead of re-selecting by reflex.
- **`reference/parent-child-devices.md`** — how parent/child devices work, useful well beyond swapping. A parent is a **device** (a multi-endpoint node fanning out to per-outlet children, shown indented in the Devices list) **or an app** (Hue, Lutron, CoCoHue, HubiThings Replica — invisible except for a "Parent app" row on the device page). Grounded on Hubitat's Parent/Child Drivers doc and its `genericComponentParentDemo.groovy` example: `addChildDevice(namespace, typeName, deviceNetworkId, properties)` from an app or a driver, built-in "Generic Component …" drivers under namespace `hubitat`, commands travelling **child → parent** via `componentXyz(cd)` with the child as first parameter, and events travelling **parent → child** via `child.parse([[name:…, value:…]])` with a List of Event maps — not `sendEvent` on the parent. Parent/child is also the only way two devices on a hub can see each other at all. Deriving a child DNI from the parent's is the common pattern (verified live: `1A` → `1A-CH1`…`CH5`) but explicitly **not** a requirement.

### Fixed

- **`hub_device_usage.py --name` could not resolve a child device** (caught by Copilot on PR #21, and a direct consequence of the tree finding below). `resolve_device_id` iterated `devices[]` at the top level only, and a child appears *only* nested in its parent's `children[]` — so `--name "Zooz Power Strip - CH1"` raised *"no device named … on this hub — check the exact display name"*, confidently wrong about a name that was exactly right. It failed precisely in the child-device case `device-migration` centres on, where `--name` is the natural handle and Swap Device is unavailable. Resolution now walks the whole forest (`_walk_devices`), so a child, a grandchild, and a name colliding across depths all resolve — the collision still raising the ambiguity error rather than silently preferring the top-level hit.
- **`/hub2/devicesList` was documented as a flat list; it is a tree.** `reference/endpoints.md` described it as `{devices:[{key, data}]}`, omitting that each entry carries `children[]`, `parent`, and `child`, that `parent`/`child` are bools meaning "**is** a parent/child" rather than pointers to one, and that **children appear only nested, never at the top level**. Code iterating `devices[]` flat silently misses every child device: the grounded hub returns 151 top-level entries and 5 children reachable only by recursing — 156 devices in all.
- **Two UI routes went undocumented.** `reference/endpoints.md` now carries them: the Settings page is **`/hub/edit`** (`/hub/settings` 404s), and `/installedapp/direct/<builtInAppType>` opens a built-in app by redirecting to a *transient* instance at `/installedapp/configure/<newId>/mainPage` — it takes the next app id, and its **Cancel** discards it (`statusJson` then returns `{}` and it never reaches `/hub2/appsList`). Swap Device is reached this way, so it is a built-in **app**, not a device page. Swapping a device's references is added to the UI-only operation list in `rules/ui-automation.md` and `reference/playwright-ui.md`.
- **A Vue-wrapped `<select>` ignores `browser_select_option`** (`reference/playwright-ui.md`). On Settings → Swap Device, selecting the "old" device that way sets the native select's value while the app never reacts — the dependent "new" picker stays disabled and empty, so the value just written reads back as though it took. The existing no-synthetic-events rule held; the read-back trap is now named.

Per `plugin-evals` Coverage the lift was measured before the scenario was committed, not after.

## 0.1.7 — 2026-07-16

### Added

- **The command path is now a first-class axis of mesh diagnosis.** Grounded on a live outage: every Z-Wave *and* Zigbee actuator on one hub (22 shades, lamps, an outlet) went dead for 13.7 h while sensors kept reporting — and `hub_mesh.py` said `critical: 0, warnings: 0`. The cause was not a radio at all. The hub's hub-mesh peer record for its companion hub still held an IP from the owner's *previous house's subnet*, so commands crossing the mesh were accepted, logged by the app, and silently dropped. The hub reported that dead peer as `active: true, offline: false, warning: null` with `lastActive` ticking every few seconds. No radio metric can see this class of fault, so `hub_mesh.py` now analyzes `GET /hub2/hubMeshJson` (newly documented) beside the two radios: peers are probed for reachability and identity (`hubUID` in `/hub/details/json` is the same identifier as `hubId`, verified across three hubs — so a reassigned address is caught as well as a dead one), and `hub_mesh.problems[]` rolls into `summary.critical`. Replaying the outage's captured bytes through the new code yields `critical: 1 / peer_unreachable` where the shipped code said all-clear. Each problem carries `shared_device_count`, the blast radius of a re-add — the asymmetry mattered live: the broken link shared 3 devices (cheap to re-add) while the *other* direction shared 148, whose removal would have unbound every app using them.
- **Z-Wave node staleness** (`age_seconds`, `stalest`, `never_heard`), the Z-Wave counterpart of `zigbee.stalest`, and the signal that actually cracked the outage. `lastTime` means "when the hub last heard this node"; for a device that only transmits *after* being commanded, that is "when a command last landed". Ranked, never flagged — silence is not a fault, and reading it needs context the script does not have. `never_heard` catches nodes the hub has never heard at all, which are reported `nodeState: OK` and pass every FAILED-keyed check (6 live on the Devices hub, including a repeater carrying 12 nodes' routes).
- **The actuator-vs-reporter split** in the `mesh-health` skill and the `zwave-zigbee-mesh` rule — the reasoning the outage turned on, and the diagnostic worth more than any single flag. Reporters (lock, motion, presence) transmit on their own, so fresh means the radio works; actuators (shade, outlet, lamp) transmit only when commanded. Reporters fresh + actuators frozen ⇒ the *command* path is broken, not the radio — and when it spans both radios at once, no radio fault can explain it, because two radios do not fail in the same second. A cluster of actuator timestamps in one narrow window is the last moment commands worked (here, a 53-second sunset routine, the outage's high-water mark).

### Fixed

- **`mesh-health` reported a confident false all-clear.** Step 2 said that if `summary.critical` and `summary.warnings` were both 0, "report the mesh looks healthy and finish" — so against 22 dead actuators the skill was engineered to stop the operator looking. Zero counters no longer license an all-clear: a new Step 4 splits the staleness rankings by actuator vs reporter, and only a run that survives *that* may report healthy. The rule and skill now open by stating that a clean radio is not a working device, since every radio metric describes a path commands do not travel.
- **A Zigbee-only run no longer warns about Z-Wave timestamps** (PR #18 review). `/hub/details/json` exists only to date Z-Wave's naive stamps — Zigbee's `lastActivity` carries its own offset — but it was fetched unconditionally, so `--radio zigbee` against a firmware lacking it raised a `fetch_warning` about node ages that run never computes. Since the skill reads any fetch warning as a blind axis blocking an all-clear, an irrelevant warning is a *false* blind axis. The fetch is gated on the Z-Wave path.
- **A peer answering 200 with JSON but no `hubUID` read as identity-verified** (PR #18 review) — `hubId: None` with `error: None` is the one shape that must never look checked-and-fine. It now carries an explicit identity-unverified error like the other unverified cases.
- **The peer probe no longer conflates reachability with identity** (caught independently by both PR #18 reviewers). `probe_peer` treated any `HubError` from `/hub/details/json` as `reachable: false` — but that endpoint is itself undocumented and version-sensitive, so a peer answering HTTP without it would have rolled a false `peer_unreachable` **critical** against a perfectly healthy peer. The two are now separate questions: `reachable: false` is reserved for a connection-level failure (nothing answered), while an HTTP response lacking or not parsing `hubUID` is reachable-with-identity-unverified, which `analyze_hub_mesh` reads as no evidence rather than as a mismatch. Re-verified against the real dead address from the outage: still `critical: 1 / peer_unreachable`.
- **Script runtime floor raised to Python 3.9+** (`scripts/README.md`), which `zoneinfo` requires and the previous documented 3.8+ contract would have broken at import. Guarding the import and degrading to UTC was the alternative and is worse than a clear ImportError: it silently overstates every zwaveJS node's age by the hub's UTC offset — precisely the bug the zone lookup exists to fix. Python 3.8 reached end-of-life on 2024-10-07 and CI runs 3.11. The stale `3.8-compatible` comment in `hub_mesh.py` is corrected to match.
- **Auxiliary endpoints no longer sink the radio diagnostics** (caught by the OpenAI policy reviewer on PR #18 as an `error-handling` Graceful Fallback violation). `/hub2/hubMeshJson` and `/hub/details/json` are undocumented and version-sensitive, and the first cut fetched them in the same fatal `try` as the radios — so a hub or firmware lacking either returned nothing at all, though `analyze()` already models `hub_mesh=None` and `naive_zone(None)` already has a fallback. Non-zero exit is now reserved for the requested radio endpoints; an auxiliary miss degrades the affected key to null and lands in a new `fetch_warnings[]` (and stderr) naming its own `consequence`. Degraded loudly, never silently: a null `hub_mesh` leaves a command-path fault unruled-out, so the skill treats a fetch warning as a blind axis rather than an all-clear. `main()` now takes `transport=` for injection, matching `hub_device_usage.py`.
- **`never_heard` misclassified unparseable timestamps** (caught by Copilot on PR #18). It keyed on `age_seconds is None`, which `parse_ts` also returns for a `lastTime` that is *present but malformed* — inventing "the hub has never heard this node" out of a parse failure, and inflating warnings now that `never_heard` counts. It keys on the raw field's absence; a present-but-unparseable stamp surfaces separately as `unparsed_timestamps[]`, a data anomaly rather than a mesh fault.
- **Flagged Z-Wave warnings never reached the rollup.** `analyze()` counted only `failed` and `packet_errors`, so `weak_signal_heuristic` (and the new `never_heard`) were computed, returned, and dropped from `summary.warnings` — a hub with routes flagged at/below the noise floor still rolled up `warnings: 0`, which the old Step 2 read as healthy. A second false all-clear, independent of the hub-mesh one and the same bug class. Found while verifying the claim that the legacy RSSI floor went unflagged: it does not — the floor works and had flagged two live nodes at `-4dB` and `0dB`; the rollup was throwing their verdict away. Live on the Devices hub, `warnings` goes 0 → 8.
- **Naive `lastTime` stamps were read as UTC, aging every zwaveJS node by the hub's offset** — a third backend trap beside the `lwrRssi` scale. The legacy backend stamps `lastTime` with an explicit `+0000` (true UTC); the zwaveJS backend emits a *naive* stamp in the hub's *local* zone. Measured live: a 70-second-old node read as 5.02 h on `America/Chicago`. Left unfixed, the staleness feature above would have been fiction on half the fleet. `parse_ts` now takes the zone, fetched from the hub's own `timeZone` rather than assumed from whatever machine the analyzer runs on.

## 0.1.6 — 2026-07-16

### Added

- **Device-removal safety** — a device delete is effectively irreversible, and a replacement gets a new device id that strands every prior reference. New always-on `device-lifecycle` rule, `device-removal` skill, and `scripts/hub_device_usage.py` (stdlib, unit-tested) read the hub's own computed "in use by" list from `GET /device/fullJson/<id>` (grounded live on 2.5.1.128) and split it into load-bearing (enabled apps) vs inert (disabled apps, idle monitors), plus dashboards, `parentApp`, and child devices — the full blast radius. The workflow: enumerate → warn with the concrete list before deleting → the delete is a hub-UI + physical action the tooling confirms, never triggers → verify the references actually cleared (auto-pruning is not guaranteed for every reference type) → on a replacement, capture the old memberships first and re-wire them onto the new id, reporting what was restored vs left. Name→id resolution lives in the script (`--name`, exact match, fails clearly on zero/multiple). ## 0.1.4 — 2026-07-16

### Added

- The **hub-receiver-noise** diagnostic, now validated end to end: the rule and `mesh-health` skill teach that an elevated or spiky **hub** noise floor with `hub_snr` well below `dest_snr` (from the `transmit_report` rollup) means the hub's own receiver is the bottleneck — its RF environment, not the device or distance — and the fix is to get the receiver out of the noise (relocate, external antenna, separate co-located 900 MHz hubs), never to touch the device.
